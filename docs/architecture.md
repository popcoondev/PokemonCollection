# アーキテクチャメモ

## 目的

このドキュメントは、現在の実装方針と今後の拡張方針を残すためのものです。
特に画像描画まわりは同期実装から段階的に再設計しているため、責務分離の意図を明文化します。

## 現在の全体構成

- `src/main.cpp`
  - アプリ状態の管理
  - タッチ入力の解釈
  - 画面モード遷移
  - UI 再描画の起点
- `src/UIController.cpp`
  - 画面全体のレイアウト描画
  - `LGFX_Sprite` を使ったメインキャンバス描画
  - 画像以外の UI パーツ描画
- `src/DataManager.cpp`
  - `index.json` の読込
  - 個別詳細 JSON の読込
- `src/ImageLoader.cpp`
  - SD 上の画像パス解決
  - PNG サイズ取得
  - PNG 描画

## 描画の責務

描画責務は今後この原則で統一します。

- UI タスクだけが画面へ反映する
- worker タスクは SD 読込と PNG デコードだけを担当する
- worker タスクは `Display` を直接触らない
- UI と worker の受け渡しは request/result キューで行う
- 古い結果は generation で破棄する

## 画像描画パイプライン

### 現在 main に入っている範囲

`すがた` タブは新しい画像パイプラインに移行済みです。

流れ:

1. UI タスクが文字や枠などの軽量 UI を先に描画する
2. `すがた` 画像が必要なら request をキューへ積む
3. worker タスクが別スプライト上で PNG を生成する
4. worker タスクが result を返す
5. UI タスクが result を受けて画像領域だけ反映する

この構成により、ページ送り時に本文を先に表示し、画像だけ後追いで出せるようにしています。

### 今後同じ方式へ載せ替える対象

- プレビュー画面
- 進化タブの画像
- 将来的な先読みキャッシュ

## generation 管理

非同期結果の取り違えを防ぐため、画像 request には generation を持たせます。

- 新しい request を積むたびに generation を更新する
- worker は generation を結果に含めて返す
- UI は現在の generation と一致した結果だけ反映する

これにより、ページ送り連打時でも古い画像の割り込みを避けます。

## キャッシュ方針

将来的には、画像パイプラインの上にキャッシュを乗せます。

優先順位:

1. 現在表示中の画像
2. 左右ページで直近に必要な画像
3. 進化タブで表示候補になる画像
4. プレビューで使う画像

基本方針:

- request 前にキャッシュを確認する
- ヒット時は即表示する
- ミス時だけ worker に decode を依頼する
- 先読みは low priority job として扱う

## 画面モード

現在の主要モード:

- `SCREEN_DETAIL`
- `SCREEN_SEARCH`
- `SCREEN_PREVIEW`

原則:

- 入力確定後は旧画面を再描画しない
- 状態を先に切り替え、その新状態に対して 1 回だけ UI を描く
- 重い画像読込は後続の非同期 job に逃がす

## 今後の更新ルール

このドキュメントは、以下の変更が入る PR で更新します。

- 新しい画面モードを追加したとき
- 画像描画パイプラインの責務を変更したとき
- キャッシュ戦略を追加または変更したとき
- UI/worker の境界を変更したとき
- 既存の同期描画を非同期へ載せ替えたとき

更新時は最低でも以下を揃えます。

- 対象機能
- 変更前の責務
- 変更後の責務
- 今後の拡張にどう繋がるか
